# ТЗ для фронтенда: чат, избранное, семейный календарь, хронология лечения

Добавлены API для четырёх блоков. Базовый URL и авторизация — как в [API-MOBILE.md](./API-MOBILE.md) (Bearer JWT).

---

## 1. Чат родитель ↔ специалист

Один общий чат на пару (родитель + специалист). Сообщения можно привязать к брони (`bookingId`).

### Родитель (роль Parent)

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/chat/conversations` | Список чатов. Ответ: `{ id, specialistUserId, lastMessageAt, unreadCount }[]` |
| GET | `api/parent/chat/conversations/{conversationOrSpecialistId}/messages?skip=0&take=50` | Сообщения. В пути можно передать **conversationId** (guid) или **specialistUserId** (string). Ответ: `{ id, senderUserId, text, bookingId?, createdAtUtc, readAtUtc? }[]` |
| POST | `api/parent/chat/conversations/{specialistUserId}/messages` | Отправить сообщение. Body: `{ "text": "string", "bookingId": "guid?" }`. Ответ: `{ id, createdAtUtc }`. Чат создаётся при первом сообщении. |
| POST | `api/parent/chat/conversations/{conversationOrSpecialistId}/read` | Отметить сообщения специалиста как прочитанные. |

### Специалист (роль Specialist)

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/specialist/chat/conversations` | Список чатов. Ответ: `{ id, parentUserId, lastMessageAt, unreadCount }[]` |
| GET | `api/specialist/chat/conversations/{conversationOrParentId}/messages?skip=0&take=50` | Сообщения. В пути: **conversationId** (guid) или **parentUserId** (string). |
| POST | `api/specialist/chat/conversations/{parentUserId}/messages` | Отправить сообщение. Body: `{ "text": "string", "bookingId": "guid?" }`. Чат должен уже существовать (родитель первый пишет). |
| POST | `api/specialist/chat/conversations/{conversationOrParentId}/read` | Отметить сообщения родителя как прочитанные. |

**UI:** экран чата с списком диалогов и экран переписки по выбранному диалогу. Поле ввода + кнопка отправить. Отображение `senderUserId` — по текущему пользователю показывать «я», иначе — имя/аватар второй стороны (подтянуть из профиля по `specialistUserId` / `parentUserId`). Непрочитанные — по `unreadCount` и `readAtUtc`.

## 2. Избранные специалисты

Родитель добавляет специалистов в избранное (одна сущность на пару родитель–специалист).

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/favorites` | Список избранных. Ответ: массив объектов как карточка специалиста: `userId`, `fullName`, `city`, `pricePerHour`, `about`, `specializations[]`, `skills[]`, `hasAvatar`. |
| POST | `api/parent/favorites/{specialistUserId}` | Добавить в избранное. 204 без тела. |
| DELETE | `api/parent/favorites/{specialistUserId}` | Удалить из избранного. 204 или 404. |
| GET | `api/parent/favorites/{specialistUserId}` | Проверить, в избранном ли. Ответ: `{ "isFavorite": true \| false }`. |

**UI:** на карточке специалиста кнопка «В избранное» / «В избранном» (иконка сердечка и т.п.). Отдельный экран/вкладка «Избранное» — список избранных специалистов (те же карточки, что в каталоге).

## 3. Семейный календарь

Общий календарь для всех опекунов семьи: брони (кто ведёт ребёнка), приёмы лекарств и сессии (кто внёс запись). Опекун видит события по тем же детям, что и родитель.

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/calendar/events?fromUtc=...&toUtc=...&childId=...` | События за период. **fromUtc**, **toUtc** — обязательные (DateTime UTC). **childId** — опционально, фильтр по одному ребёнку. |

**Формат события (единый массив, тип по полю `type`):**

- **booking:** `type: "booking"`, `id`, `childId`, `specialistUserId`, `startsAtUtc`, `endsAtUtc`, `status`, `assignedCaregiver` (объект `{ email, relation }` или null).
- **med_intake:** `type: "med_intake"`, `id`, `childId`, `drug`, `dose`, `timeUtc`, `taken`, `recordedByUserId`.
- **session:** `type: "session"`, `id`, `childId`, `sessionType`, `durationMin`, `timeUtc`, `recordedByUserId`.

**Кто ведёт на встречу:** при создании брони можно передать `assignedCaregiverMemberId` (см. [API-MOBILE.md](./API-MOBILE.md), раздел «Бронирования»). Позже можно обновить назначение:

| Метод | URL | Описание |
|-------|-----|----------|
| PATCH | `api/parent/bookings/{id}/assign-caregiver` | Назначить опекуна. Body: `{ "assignedCaregiverMemberId": "guid?" }`. Пустой/null — снять назначение. |

**UI:** календарь (неделя/месяц) с событиями трёх типов. Для брони — подпись «Ведёт: имя/email опекуна» из `assignedCaregiver`. Для медикаментов и сессий — при наличии `recordedByUserId` показывать «Внёс: …» (имя по userId, если есть в профилях/опекунах). Фильтр по ребёнку — выпадающий список детей.

## 4. Полная хронология лечения ребёнка

Единая лента по ребёнку: заметки, документы, брони, итоги встреч (outcome), записи дня, приёмы лекарств, сессии, инциденты. Для специалиста доступ только если есть/была бронь с этим ребёнком.

### Родитель

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/children/{childId}/chronology?fromUtc=...&toUtc=...` | Хронология по ребёнку. **fromUtc**, **toUtc** — опционально, фильтр по дате. |

### Специалист

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/specialist/children/{childId}/chronology?fromUtc=...&toUtc=...` | Та же хронология. 404, если у специалиста нет ни одной брони с этим ребёнком. |

**Формат элемента ленты (все в одном массиве, сортировка по `atUtc`):**

| type | Поля | Описание |
|------|------|----------|
| `note` | id, atUtc, summary | Заметка по ребёнку. |
| `document` | id, atUtc, summary (имя файла) | Документ. |
| `booking` | id, atUtc, specialistUserId, status, summary | Бронь. |
| `outcome` | id, bookingId, atUtc, summary | Итог встречи (после закрытия брони специалистом). |
| `daily_entry` | id, atUtc, date, summary | Запись дня (трекер). |
| `med_intake` | id, atUtc, summary (препарат, доза, принято/нет) | Приём лекарства. |
| `session` | id, atUtc, summary (тип, длительность) | Сессия. |
| `incident` | id, atUtc, intensity, summary | Инцидент. |

**UI:** экран «Хронология» по выбранному ребёнку: таймлайн (или список) с карточками по типу. Родитель открывает из профиля ребёнка. Специалист — из контекста брони/карточки ребёнка (если есть доступ). Даты в формате `atUtc` (ISO 8601), отображать в локальной таймзоне пользователя.

## 5. Изменения в существующих API

- **Создание брони** `POST api/parent/bookings`: в теле можно передать **assignedCaregiverMemberId** (guid опекуна из своей семьи) — кто ведёт на встречу.
- **Ответ брони** (родитель и специалист): в объекте брони добавлено поле **assignedCaregiverMemberId** (guid?).
- При создании записи **приёма лекарства** и **сессии** в трекере бэкенд сам проставляет **recordedByUserId** (текущий пользователь); в календаре это поле отдаётся в событиях `med_intake` и `session`.

## 7. Семейный режим (роли, участники, контекст семьи)

Бэкенд переведён на **семейный контекст**: все данные родителя (профиль, дети, брони, чат, избранное, календарь, трекер, хронология) привязаны к **семье** (одному ParentProfile). В семье есть **владелец** (owner) и **участники** (caregivers): админы и обычные опекуны. Участник видит те же детей, календарь и трекер; чат и избранное — общие на семью.

### 7.1. Текущая семья и роль

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/family` | Текущая семья и моя роль. Всегда 200. |

**Ответ, если пользователь не в семье** (нет профиля и нет принятого приглашения):

```json
{ "inFamily": false, "message": "Create or join a family first (create profile or accept invite)" }
```

**Ответ, если пользователь в семье**:

```json
{
  "inFamily": true,
  "myRole": "owner" | "admin" | "caregiver",
  "parentProfileId": "guid",
  "ownerUserId": "string",
  "ownerEmail": "string?",
  "ownerFullName": "string?",
  "canManageMembers": true | false,
  "members": [
    { "id": null, "email": "...", "userId": "...", "relation": null, "isAdmin": true, "status": 2, "invitedAtUtc": "...", "acceptedAtUtc": "...", "role": "owner" },
    { "id": "guid", "email": "...", "userId": "..." | null, "relation": "...", "isAdmin": true|false, "status": 0|1|2|..., "invitedAtUtc": "...", "acceptedAtUtc": "..." | null, "role": "admin" | "caregiver" }
  ]
}
```

- **myRole:** `owner` — владелец профиля (создал семью), `admin` — опекун с правом управлять участниками, `caregiver` — опекун без прав управления.
- **canManageMembers:** `true` только у owner и admin. Ими показывать кнопки пригласить/удалить/изменить роль.
- **members:** первый элемент — владелец (role `owner`, `id` null), остальные — опекуны из `api/parent/caregivers`. **status:** 0 = Pending, 1 = Active, 2 = Accepted, 3 = Revoked (см. контракт бэкенда).

### 7.2. Управление участниками (опекунами)

Эндпоинты те же, что раньше; права проверяются по **canManageMembers** (owner/admin):

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `api/parent/caregivers` | Список участников семьи. 404, если нет семьи. |
| POST | `api/parent/caregivers/invite` | Пригласить по email. Body: `{ "email": "...", "relation": "...", "isAdmin": false }`. 403, если нет права. |
| PUT | `api/parent/caregivers/{memberId}` | Обновить (relation, isAdmin, status). 403, если нет права. |
| DELETE | `api/parent/caregivers/{memberId}` | Удалить из семьи. 403, если нет права. |
| POST | `api/parent/caregivers/{memberId}/accept` | Принять приглашение (вызов от пользователя с email приглашённого). |

После принятия приглашения у пользователя появляется семья (`GET api/parent/family` → `inFamily: true`, `myRole: "caregiver"` или `"admin"`).

### 7.3. Поведение остальных parent-эндпоинтов

- **Профиль:** `GET api/parent/profile` — возвращает профиль **семьи** (владельца). 404, если нет семьи. `POST api/parent/profile` (upsert) — только **owner** может редактировать; caregiver получает 403.
- **Дети, брони, чат, избранное, календарь, трекер, хронология:** работают в контексте семьи (общие данные). Если у пользователя нет семьи (нет профиля и не принял приглашение), эти эндпоинты возвращают **404** с телом вида `{ "error": "No family" }` или `{ "error": "No family. Create profile or accept invite." }`.
- **Добавление ребёнка** `POST api/parent/children`: любой участник семьи (owner/admin/caregiver) может создать ребёнка в этой семье. 404, если нет семьи.

### 7.4. Рекомендации для UI

1. **При первом заходе под родителем** вызвать `GET api/parent/family`. Если `inFamily: false` — показать онбординг: «Создайте профиль» (форма → `POST api/parent/profile`) или «Вам отправили приглашение — примите его» (экран принятия по ссылке/коду или список приглашений, если появится отдельный эндпоинт).
2. **Экран «Семья» / «Участники»:** список из `members`; у владельца подпись «Владелец», у опекунов — роль и статус. Если `canManageMembers === true` — кнопки пригласить, для каждого участника (кроме owner) — изменить роль / удалить.
3. **При 404 с `error: "No family"`** на любом parent-эндпоинте — перенаправить на онбординг (создать профиль или принять приглашение).
4. **Редактирование профиля:** кнопку «Редактировать профиль» показывать только при `myRole === "owner"` (или скрывать для caregiver, т.к. бэк вернёт 403).

---

Итог для фронта: четыре блока (чат, избранное, календарь, хронология) + доработки брони и трекера + **семейный режим** (GET family, роли, управление участниками, обработка 404 «No family»). Если нужны примеры запросов/ответов по конкретному эндпоинту — можно расписать отдельно.
